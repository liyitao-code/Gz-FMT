# 新蜕变关系设计提案（v2 — 基于范式的方法）

## 背景与动机

### 现有蜕变关系评估

当前已实现 5 种旧蜕变关系 + 2 种新蜕变关系：

| # | 测试名称 | 蜕变关系 | 是否依赖 reset | 范式 |
|---|---------|---------|:------------:|------|
| 1 | Motion Test | 位移 = 速度 × 时间 | 是 | Reset-Compare |
| 2 | Force Additivity Test | F1+F2 = F_total | 是 | Reset-Compare |
| 3 | Time Scaling Test | RTF 不影响物理 | 是 | Reset-Compare |
| 4 | Mass Scaling Test | F=ma → d1/d2 = m2/m1 | 是 | Reset-Compare |
| 5 | Symmetry Test | 不同方向同等力 → 同等位移 | 是 | Reset-Compare |
| 6 | Determinism Test | 两次独立运行 → 结果一致 | **否** | Cross-Process |

### 核心发现：结构性盲区

**除了 Determinism Test，所有测试共享同一执行范式：**

```
Phase A: 施力 → 推进仿真 → 记录结果
         ↓
       reset_simulation()
         ↓
Phase B: 施力（变量改变）→ 推进仿真 → 记录结果
         ↓
       比较 A vs B
```

**后果：** 测试的物理性质（力可加性、质量缩放、对称性等）不同，但**故障检测能力高度重叠**——本质上都在测试"reset 是否正确"。

### 已发现的 bug 类型（全部是 reset 相关）

- `gz-sim-buoyancy-system`：reset 后浮力失效（不实现 `ISystemReset`）
- `gz-sim-wind-effects-system`：reset 后风力失效（同类根因）

### 无法检测的 bug 类型

1. 幽灵力/能量注入（无力施加时模型异常运动）
2. 力的隔离性问题（施力到 A，B 也受影响）
3. 力的生命周期 bug（clear_wrench 后力仍然存在）
4. 求解器不稳定性（位移非单调、突变、传送）
5. 非确定性 bug（竞态条件、未初始化变量）——Determinism Test 可检测

---

## 核心思想：按范式而非物理性质设计

**关键洞察：测试的结构范式决定了它能检测的 bug 类别。**

旧方法：每个测试验证不同的物理定律，但用相同的范式执行 → 检测同一类 bug。
新方法：每个测试使用不同的结构范式 → 检测不同类别的 bug。

### 范式分类

| 范式 | 描述 | 可检测的 bug 类别 |
|------|------|-----------------|
| **Reset-Compare**（旧） | A → reset → B → 比较 | reset 相关 bug |
| **A: Cross-Process**（已实现） | 两个独立 Gazebo 进程比较 | 非确定性、竞态条件 |
| **B: Single-Run Invariant**（新） | 单次运行中检查物理不变量 | 幽灵力、能量注入、求解器发散 |
| **C: Same-Run Multi-Model**（新） | 同一运行中比较不同模型 | 力泄漏、实体隔离、目标错误 |
| **D: Sequential No-Reset**（新） | 同一运行分阶段，无 reset | 力持续性、状态泄漏、清理 bug |

## 新蜕变关系提案

### 提案 B1：零输入稳定性测试（Zero-Input Stability Test）

**优先级：P0（最高）**
**范式：B — Single-Run Invariant**

#### 蜕变关系

> 不施加任何外力 → 模型在水平方向（x, y）不应产生显著位移

#### 核心思想

所有现有测试都**施加力**然后检查结果。施加的力掩盖了模型本身可能存在的异常运动。本测试**不施加任何力**，纯粹检查模型是否在无外力条件下保持静止。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 暂停仿真，不施加任何力
3. step_simulation(5000)  — 推进 5 秒
4. 获取位置 P1
5. 检查：|P1.x - P0.x| < 0.05m AND |P1.y - P0.y| < 0.05m
   （允许 z 轴变化，因为重力可能导致模型下沉/稳定）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 插件幽灵力 | 风力/浮力插件在不该生效时产生力 |
| NaN/inf 传播 | 数值异常导致模型随机移动 |
| 初始化速度残留 | 模型创建时带有非零速度 |
| 求解器能量注入 | 接触求解器在稳态时注入能量 |
| 插件初始化脉冲 | 插件启动时产生短暂的力 |

#### 关键区别

所有现有测试施加力后检查结果。施加的力（通常几百 N）完全掩盖了可能存在的小幽灵力（几 N）。本测试不施加力，因此任何异常运动都能被检测到。

### 提案 C1：多模型力隔离测试（Multi-Model Force Isolation Test）

**优先级：P0（最高）**
**范式：C — Same-Run Multi-Model**

#### 蜕变关系

> 仅对模型 A 施力 → 无接触的远处模型 B 不应受影响

#### 核心思想

所有现有测试只操作和观察一个模型。如果力的应用系统存在实体路由 bug（力被施加到错误的模型），单模型测试永远检测不到。本测试在施力到目标模型的同时，监控旁观者模型是否被错误影响。

#### 测试流程

```
1. 获取所有可测试模型
2. 选择两个模型：M_target（施力目标）和 M_bystander（旁观者）
3. 验证两模型距离足够远（无接触可能）
4. 记录初始位置：P_target_0, P_bystander_0
5. 对 M_target 施力 F
6. step_simulation(N)
7. 获取位置：P_target_1, P_bystander_1
8. 检查：M_target 有显著位移（力确实生效，sanity check）
9. 检查：|P_bystander_1.xy - P_bystander_0.xy| < 0.05m（旁观者未被影响）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 力目标路由错误 | EntityWrench 实体名/ID 解析错误 |
| 全局力泄漏 | apply-link-wrench 系统对所有模型施力 |
| 插件间干扰 | 一个模型的插件状态影响另一个 |
| 共享状态损坏 | 模型实例之间共享了不该共享的状态 |

#### 关键区别

完全不使用 reset。在**同一次仿真运行**中同时观察两个模型。测试的是力的**隔离性**而非力的**正确性**。

### 提案 D1：撤力响应测试（Force Removal Response Test）

**优先级：P1**
**范式：D — Sequential No-Reset**

#### 蜕变关系

> 施加持续力后撤除 → 撤力后速度不应增加（应减速或匀速）

#### 核心思想

现有测试从不检查 `clear_model_wrench()` 是否真正生效。本测试分为两个连续阶段（无 reset）：施力阶段和撤力阶段。如果撤力后加速度仍为正，说明力没有被正确清除。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 施加持续力 F（+x 方向）
3. step_simulation(2000)  — 施力阶段 2 秒
4. 记录位置 P1
5. clear_model_wrench(M)  — 撤除力
6. step_simulation(1000)  — 滑行阶段 1 秒
7. 记录位置 P2
8. step_simulation(1000)  — 继续滑行 1 秒
9. 记录位置 P3

速度估算：
  v_force = (P1.x - P0.x) / 2.0    — 施力阶段平均速度
  v_coast1 = (P2.x - P1.x) / 1.0   — 滑行第 1 秒平均速度
  v_coast2 = (P3.x - P2.x) / 1.0   — 滑行第 2 秒平均速度

检查：
  - v_coast1 <= v_force * 1.1   （滑行速度不超过施力期间平均速度）
  - v_coast2 <= v_coast1 * 1.05 （滑行速度不应增加）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| clear_wrench 未生效 | wrench/clear 话题处理 bug |
| 力持续性泄漏 | 持续力在 clear 后仍然存在 |
| 插件状态残留 | 插件在力撤除后保持内部累积状态 |
| wrench 系统记账错误 | 对错误实体的力在 clear 后持续 |

#### 关键区别

两个阶段在**同一次运行中连续执行**，完全不使用 reset。测试的是力的**生命周期管理**（apply → clear → verify cleared），这是其他所有测试都不涉及的。

### 提案 B2：时序单调性测试（Temporal Monotonicity Test）

**优先级：P1**
**范式：B — Single-Run Invariant**

#### 蜕变关系

> 恒定力持续作用 → 力方向上的位移必须随时间单调递增

#### 核心思想

所有现有测试只采样**起点和终点**。如果模型在中间发生了异常（突然回弹、位置跳变、力间歇性失效），终点看起来可能仍然正常。本测试在多个时间点采样，检查位移轨迹的单调性和连续性。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 施加持续力 F（+x 方向）
3. 循环 K=10 次（每次 500 步 = 0.5 秒）：
   - step_simulation(500)
   - 记录位置 P_i
4. 计算 x 方向位移序列：d_i = P_i.x - P0.x

检查：
  - 单调性：d_1 < d_2 < d_3 < ... < d_K （严格递增）
  - 平滑性：|d_{i+1} - d_i| < 3 × |d_i - d_{i-1}|（无突变/传送）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 力间歇性失效 | 插件偶尔跳过一步不施力 |
| 求解器发散 | 位置突然跳变 |
| 碰撞反弹传送 | 碰撞响应将模型弹回 |
| 时间步进累积误差 | 积分误差导致位移非单调 |
| 力方向翻转 | 力的方向在某些条件下反转 |

#### 关键区别

唯一一个检查**轨迹形状**的测试。所有其他测试只比较两个端点。间歇性 bug（偶尔出错几步然后恢复正常）会通过所有端点测试但无法通过本测试。

## 旧提案重新评估

以下提案来自 v1 版本。经过范式分析后的重新评估：

### 已实现但重新定级

| 提案 | 原优先级 | 新评估 | 原因 |
|------|---------|--------|------|
| Determinism Test | P0 | **P0 保持** | 唯一的 Cross-Process 范式，检测独特的 bug 类型 |
| Symmetry Test | P0 | **降级为旧范式** | 本质是 Reset-Compare，与旧测试检测同类 bug |

### 旧提案保留为候选（P2）

| 提案 | 范式 | 备注 |
|------|------|------|
| Gravity Scaling Test | Reset-Compare → 可重构为 Cross-Process | 测试重力系统，但需要重构范式才有独特价值 |
| Torque Scaling Test | Reset-Compare | 测试旋转动力学，仍为 Reset-Compare |
| Translation Invariance Test | Reset-Compare → 可重构为 Same-Run Multi-Model | 测试数值精度，可用 C 范式重构 |
| Collision Momentum Conservation | Single-Run Invariant | 原本就是 B 范式，实现复杂度高 |
