# 新蜕变关系设计提案（v2 — 基于范式的方法）

## 背景与动机

### 现有蜕变关系评估

当前已实现 5 种旧蜕变关系 + 2 种新蜕变关系：

| # | 测试名称 | 蜕变关系 | 是否依赖 reset | 范式 |
|---|---------|---------|:------------:|------|
| 1 | Motion Test | 位移 = 速度 × 时间 | 是 | Reset-Compare |
| 2 | Force Additivity Test | F1+F2 = F_total | 是 | Reset-Compare |
| 3 | Time Scaling Test | RTF 不影响物理 | 是 | Reset-Compare |
| 4 | Mass Scaling Test | F=ma → d1/d2 = m2/m1 | 是 | Reset-Compare |
| 5 | Symmetry Test | 不同方向同等力 → 同等位移 | 是 | Reset-Compare |
| 6 | Determinism Test | 两次独立运行 → 结果一致 | **否** | Cross-Process |

### 核心发现：结构性盲区

**除了 Determinism Test，所有测试共享同一执行范式：**

```
Phase A: 施力 → 推进仿真 → 记录结果
         ↓
       reset_simulation()
         ↓
Phase B: 施力（变量改变）→ 推进仿真 → 记录结果
         ↓
       比较 A vs B
```

**后果：** 测试的物理性质（力可加性、质量缩放、对称性等）不同，但**故障检测能力高度重叠**——本质上都在测试"reset 是否正确"。

### 已发现的 bug 类型（全部是 reset 相关）

- `gz-sim-buoyancy-system`：reset 后浮力失效（不实现 `ISystemReset`）
- `gz-sim-wind-effects-system`：reset 后风力失效（同类根因）

### 无法检测的 bug 类型

1. 幽灵力/能量注入（无力施加时模型异常运动）
2. 力的隔离性问题（施力到 A，B 也受影响）
3. 力的生命周期 bug（clear_wrench 后力仍然存在）
4. 求解器不稳定性（位移非单调、突变、传送）
5. 非确定性 bug（竞态条件、未初始化变量）——Determinism Test 可检测

---

## 核心思想：按范式而非物理性质设计

**关键洞察：测试的结构范式决定了它能检测的 bug 类别。**

旧方法：每个测试验证不同的物理定律，但用相同的范式执行 → 检测同一类 bug。
新方法：每个测试使用不同的结构范式 → 检测不同类别的 bug。

### 范式分类

| 范式 | 描述 | 可检测的 bug 类别 |
|------|------|-----------------|
| **Reset-Compare**（旧） | A → reset → B → 比较 | reset 相关 bug |
| **A: Cross-Process**（已实现） | 两个独立 Gazebo 进程比较 | 非确定性、竞态条件 |
| **B: Single-Run Invariant**（新） | 单次运行中检查物理不变量 | 幽灵力、能量注入、求解器发散 |
| **C: Same-Run Multi-Model**（新） | 同一运行中比较不同模型 | 力泄漏、实体隔离、目标错误 |
| **D: Sequential No-Reset**（新） | 同一运行分阶段，无 reset | 力持续性、状态泄漏、清理 bug |

## 新蜕变关系提案

### 提案 B1：零输入稳定性测试（Zero-Input Stability Test）

**优先级：P0（最高）**
**范式：B — Single-Run Invariant**

#### 蜕变关系

> 不施加任何外力 → 模型在水平方向（x, y）不应产生显著位移

#### 核心思想

所有现有测试都**施加力**然后检查结果。施加的力掩盖了模型本身可能存在的异常运动。本测试**不施加任何力**，纯粹检查模型是否在无外力条件下保持静止。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 暂停仿真，不施加任何力
3. step_simulation(5000)  — 推进 5 秒
4. 获取位置 P1
5. 检查：|P1.x - P0.x| < 0.05m AND |P1.y - P0.y| < 0.05m
   （允许 z 轴变化，因为重力可能导致模型下沉/稳定）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 插件幽灵力 | 风力/浮力插件在不该生效时产生力 |
| NaN/inf 传播 | 数值异常导致模型随机移动 |
| 初始化速度残留 | 模型创建时带有非零速度 |
| 求解器能量注入 | 接触求解器在稳态时注入能量 |
| 插件初始化脉冲 | 插件启动时产生短暂的力 |

#### 关键区别

所有现有测试施加力后检查结果。施加的力（通常几百 N）完全掩盖了可能存在的小幽灵力（几 N）。本测试不施加力，因此任何异常运动都能被检测到。

### 提案 C1：多模型力隔离测试（Multi-Model Force Isolation Test）

**优先级：P0（最高）**
**范式：C — Same-Run Multi-Model**

#### 蜕变关系

> 仅对模型 A 施力 → 无接触的远处模型 B 不应受影响

#### 核心思想

所有现有测试只操作和观察一个模型。如果力的应用系统存在实体路由 bug（力被施加到错误的模型），单模型测试永远检测不到。本测试在施力到目标模型的同时，监控旁观者模型是否被错误影响。

#### 测试流程

```
1. 获取所有可测试模型
2. 选择两个模型：M_target（施力目标）和 M_bystander（旁观者）
3. 验证两模型距离足够远（无接触可能）
4. 记录初始位置：P_target_0, P_bystander_0
5. 对 M_target 施力 F
6. step_simulation(N)
7. 获取位置：P_target_1, P_bystander_1
8. 检查：M_target 有显著位移（力确实生效，sanity check）
9. 检查：|P_bystander_1.xy - P_bystander_0.xy| < 0.05m（旁观者未被影响）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 力目标路由错误 | EntityWrench 实体名/ID 解析错误 |
| 全局力泄漏 | apply-link-wrench 系统对所有模型施力 |
| 插件间干扰 | 一个模型的插件状态影响另一个 |
| 共享状态损坏 | 模型实例之间共享了不该共享的状态 |

#### 关键区别

完全不使用 reset。在**同一次仿真运行**中同时观察两个模型。测试的是力的**隔离性**而非力的**正确性**。

### 提案 D1：撤力响应测试（Force Removal Response Test）

**优先级：P1**
**范式：D — Sequential No-Reset**

#### 蜕变关系

> 施加持续力后撤除 → 撤力后速度不应增加（应减速或匀速）

#### 核心思想

现有测试从不检查 `clear_model_wrench()` 是否真正生效。本测试分为两个连续阶段（无 reset）：施力阶段和撤力阶段。如果撤力后加速度仍为正，说明力没有被正确清除。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 施加持续力 F（+x 方向）
3. step_simulation(2000)  — 施力阶段 2 秒
4. 记录位置 P1
5. clear_model_wrench(M)  — 撤除力
6. step_simulation(1000)  — 滑行阶段 1 秒
7. 记录位置 P2
8. step_simulation(1000)  — 继续滑行 1 秒
9. 记录位置 P3

速度估算：
  v_force = (P1.x - P0.x) / 2.0    — 施力阶段平均速度
  v_coast1 = (P2.x - P1.x) / 1.0   — 滑行第 1 秒平均速度
  v_coast2 = (P3.x - P2.x) / 1.0   — 滑行第 2 秒平均速度

检查：
  - v_coast1 <= v_force * 1.1   （滑行速度不超过施力期间平均速度）
  - v_coast2 <= v_coast1 * 1.05 （滑行速度不应增加）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| clear_wrench 未生效 | wrench/clear 话题处理 bug |
| 力持续性泄漏 | 持续力在 clear 后仍然存在 |
| 插件状态残留 | 插件在力撤除后保持内部累积状态 |
| wrench 系统记账错误 | 对错误实体的力在 clear 后持续 |

#### 关键区别

两个阶段在**同一次运行中连续执行**，完全不使用 reset。测试的是力的**生命周期管理**（apply → clear → verify cleared），这是其他所有测试都不涉及的。

### 提案 B2：时序单调性测试（Temporal Monotonicity Test）

**优先级：P1**
**范式：B — Single-Run Invariant**

#### 蜕变关系

> 恒定力持续作用 → 力方向上的位移必须随时间单调递增

#### 核心思想

所有现有测试只采样**起点和终点**。如果模型在中间发生了异常（突然回弹、位置跳变、力间歇性失效），终点看起来可能仍然正常。本测试在多个时间点采样，检查位移轨迹的单调性和连续性。

#### 测试流程

```
1. 选择模型 M，记录初始位置 P0
2. 施加持续力 F（+x 方向）
3. 循环 K=10 次（每次 500 步 = 0.5 秒）：
   - step_simulation(500)
   - 记录位置 P_i
4. 计算 x 方向位移序列：d_i = P_i.x - P0.x

检查：
  - 单调性：d_1 < d_2 < d_3 < ... < d_K （严格递增）
  - 平滑性：|d_{i+1} - d_i| < 3 × |d_i - d_{i-1}|（无突变/传送）
```

#### 能检测的 bug 类型

| Bug 类型 | 示例 |
|---------|------|
| 力间歇性失效 | 插件偶尔跳过一步不施力 |
| 求解器发散 | 位置突然跳变 |
| 碰撞反弹传送 | 碰撞响应将模型弹回 |
| 时间步进累积误差 | 积分误差导致位移非单调 |
| 力方向翻转 | 力的方向在某些条件下反转 |

#### 关键区别

唯一一个检查**轨迹形状**的测试。所有其他测试只比较两个端点。间歇性 bug（偶尔出错几步然后恢复正常）会通过所有端点测试但无法通过本测试。

## New Metamorphic Relations (v3)

### 1. Energy Consistency Test

**Idea**: Under simple, closed-world conditions, the simulation should not inject or rapidly create mechanical energy.

- **Metamorphic relation**:  
  - In a conservative scene (no external work, no active controllers), total mechanical energy \(E = T + V\) should be approximately conserved over time.  
  - In a damped scene (explicit damping), \(E\) should decrease smoothly over time, without sudden spikes.
- **Test sketch**:
  1. Pick a single model and a simple world (no plugins that actively inject forces).
  2. Record its pose and linear velocity at several equally spaced time samples during free motion.
  3. Estimate kinetic + potential energy at each sample.
  4. Check:
     - Conservative case: energy drift stays within a small tolerance and does not jump up suddenly.
     - Damped case: energy is monotonically non-increasing (within a small epsilon).

This test can reveal integrator instabilities, “energy injection” bugs in contact solvers, and inconsistent damping behaviour.

---

### 2. Time-Scaling Invariant Test

**Idea**: Halving the integration step size should not qualitatively change the outcome for the same physical duration.

- **Metamorphic relation**:  
  - Run the same scene twice for the same physical time \(T\), once with step size \(h\), once with step size \(h/2\).  
  - The final pose difference should stay below a small threshold.
- **Test sketch**:
  1. Select a world and a model.
  2. Run for duration \(T\) with `max_step_size = h` (or equivalent), record final pose \(P_h\).
  3. Reset to the same initial state.
  4. Run for the same duration \(T\) with `max_step_size = h/2`, record final pose \(P_{h/2}\).
  5. Compare \(|P_h - P_{h/2}|\) against a tolerance.

This test detects integrators that are extremely sensitive to step size or become unstable at certain step sizes.

---

### 3. Symmetry Invariant Test

**Idea**: Symmetric setups should produce symmetric outcomes.

- **Metamorphic relation**:  
  - For a model in a symmetric environment, applying the same-magnitude force along +x and +y should lead to trajectories that are symmetric up to a rotation.
- **Test sketch**:
  1. Place a model in a flat, symmetric environment (no slopes, no anisotropic fields).
  2. Apply a constant force \(F\) along +x for duration \(T\), record horizontal displacement \(d_x\).
  3. Reset to the same initial state.
  4. Apply the same-magnitude force \(F\) along +y for duration \(T\), record displacement \(d_y\).
  5. Check that \(|d_x| \approx |d_y|\) within a small tolerance.

This test can reveal directional bias or asymmetries in the contact solver or force application pipeline.

---

### 4. Contact Stability Test (Landing / Penetration Invariance)

**Idea**: After a model lands on a static contact surface, contact should be stable and not “sink” or “explode”.

- **Metamorphic relation**:
  - For a model dropped from a fixed height onto a flat plane, after some settling time, its contact depth and height should stay within a small band and not drift.
- **Test sketch**:
  1. Start with a model above a flat ground plane, with gravity enabled and no other forces.
  2. Run the simulation until after impact and a short settling period.
  3. Sample the model’s vertical position at several later times.
  4. Check that:
     - The height does not decrease monotonically (no continuous sinking into the ground).
     - The height oscillations, if any, damp out and stay within a narrow band.

This test targets contact / penetration resolution bugs and long-term contact instability.

翻译：
1. 能量一致性 / 能量单调性（Energy Consistency）
场景：只在无空气阻力、无主动控制、无外加功的情况下运行一段时间（例如只有重力 + 地面接触）。
蜕变关系：
保守力系统中，机械能应近似守恒（或只因数值误差缓慢漂移）；
有阻尼时，机械能应单调减少，不应突然暴涨。
可检测问题：
接触求解里“能量爆炸”/ 生成能量；
不合理的数值耗散 / 增益。
2. 时间缩放蜕变（Time-Scaling）
思路：同一个场景，跑两遍：
一遍用 max_step_size = h；
一遍用 max_step_size = h/2，总仿真时长不变（步数翻倍）。
期望：最终状态（位置/速度）应该在一个小误差范围内接近；
如果步长一改，结果完全不同，说明算法对时间步长极度敏感。
优点：对任何场景都适用，特别能测出“低阶/不稳定积分器”的问题。
3. 对称性测试（Symmetry Metamorphism）
构造：
把一个 world 在 x 轴上镜像，或者旋转 180°；
施加镜像后的力/初始条件。
蜕变关系：
理论上，对称变换后的轨迹应该也是镜像/旋转后的形状；
如果一边通过、一边不通过，可能有引擎中“方向性偏置”的 bug（如只在 +x 方向正确，-x 出错）。
适用对象：很多基础 demo world（车往左 vs 往右、箱子推前 vs 推后）。
4. 超位移 / 穿透检测的变体（Contact Invariance）
场景：一个模型从上方自由落到地面，和从同样高度但慢速推进到地面的两个实验。
关系：
落地后应在合理的高度/穿透深度范围内稳定，而不会越来越“踩进地里”或反复弹飞。
这可以看作 zero_input_stability 的改良版：只针对“已知会接触地面”的场景，检查接触稳定性，而不是检查“一切不应动”。

1. 关节约束稳定性测试（Joint Constraint Stability）
目标 bug 类型：
关节/约束“拉长 / 压缩”漂移（constraint drift）；
关节在长时间运动后“爆炸”或出现明显抖动；
插件 / 关节参数导致的奇怪漂移（比如 articulations、软关节）。
蜕变关系（Single-Run Invariant）：
> 对于通过关节连接的两个 link，它们之间的约束（距离 / 关节角度）在仿真过程中应该保持在一个小的容差范围内，而不是缓慢漂移或突然失真。
测试大致形态：
选一个含关节的模型（铰链、滑动关节、车轮悬挂、机械臂等）；
单次运行中：
记录若干对“约束量”：
铰链：两 link 之间的固定距离；
滑动关节：限制范围内的相对位移；
复合关节：可投影为一个或多个标量（例如 link 质心距离）。
施加一些适度的力矩 / 力，让关节在合理范围内运动；
定期采样上述“约束量”；
检查：
约束没有明显 线性漂移（比如距离越来越长）；
不出现突然的 大跳变（说明关节数值不稳定）。
区别于你现有测试：
不关心位置“对不对”，只关心“关节有没有被 solver 玩坏”；
可以在有重力、有碰撞的真实世界里跑，不那么怕假阳性。
2. 质量缩放的“单次运行”变体（Mass-Scaling Invariant, No Reset）
旧的 mass_scaling 是 Reset-Compare 范式，主要又变成了“reset 正不正确”。可以考虑一个不 reset 的版本，专门打物理层：
目标 bug 类型：
质量参与 F=ma 的实现有问题（例如质量改变后加速度不按预期变）；
质量改变后，求解器稳定性/阻尼行为异常；
插件 / 控制器里偷偷把质量写死（改质量后行为不变）。
蜕变关系（Sequential No-Reset, 范式 D）：
> 在同一运行中，先用质量 m 施加某个力得到位移 d₁，再把质量改为 k·m 并相应调整力大小（保持期望加速度不变），最终位移 d₂ 应该在一个小误差范围内接近 d₁。
测试大致形态：
同一运行里两个阶段，不 reset world：
Phase A：质量 = m，力 = F，持续时间 T，得到位移 d₁；
Phase B：质量调成 k·m，力调成 k·F（理论上 a 一样），再跑 T，得到位移 d₂；
检查：
d₁ 和 d₂ 在容差内接近；
同时监控是否出现明显不同的数值行为（抖动、爆炸等）。
价值：
真正测试的是“质量参与动力学的实装”和“质量修改的生命周期管理”，与现有的 determinism / monotonicity / force_removal 关注点不同。
3. 环境扰动鲁棒性测试（Environment Perturbation Robustness）
目标 bug 类型：
远处静态物体、与测试无关的插件，被错误地纳入计算（例如错误的广义力 / 全局字段）；
世界里加一个“看似无关”的模型，对目标模型产生不合理影响（例如重力场实现有 bug，或某个 plugin 使用了全局状态）。
蜕变关系（Same-Run or Reset-Compare，但关注“环境不变量”）：
> 在只改变“与目标模型物理上无直接关系”的环境（远处静态物体、装饰性模型、无接触的障碍）时，目标模型在同一力输入下的行为不应发生显著变化。
测试大致形态（可以有两种实现方式）：
Reset-Compare 版（简单实现）：
Phase A：当前 world，选一个 model，施力 F 跑 T 秒，记录位移 d_A；
Phase B：往场景里额外插入一个“远处静态模型 / 装饰物”，其他条件相同，再跑 T 秒，记录 d_B；
检查 |d_A − d_B| 是否在容差内。
Single-Run 近似版：
同一运行中：
先在“原环境”下施力 F 跑 T/2，得到一段轨迹；
中途插入一个远处静态模型，再施力 F 跑 T/2，看轨迹是否在插入点之后发生异常跳变。
适合暴露的 bug：
插件/系统遍历所有 Entity 时漏加过滤条件，导致“全局力”；
某些系统按 world 中实体总数/顺序改变行为；
内部使用了全局状态（例如“最后创建的模型”等）影响目标模型。
