## 二、做到哪一步了（进度概览）

### 2.1 测试框架与范式

- **旧蜕变关系**（motion, force_additivity, time_scaling, mass_scaling, symmetry, rewind）已**临时屏蔽**，因依赖 reset，且易受 wall-clock 时间、scene/info 陈旧数据等干扰，假阳性多。
- **当前在跑的 5 种测试**（新范式，均不依赖 reset）：
  - **Determinism**：两进程同 SDF/同力/同步数，比较终点位姿是否一致。
  - **Zero-Input Stability**：不施力，只推进仿真，看水平是否漂移。
  - **Force Isolation**：只对模型 A 施力，看远处模型 B 是否被带动。
  - **Force Removal**：施力 → 撤力 → 滑行，看撤力后是否还加速。
  - **Temporal Monotonicity**：恒力下多点采样位移，看是否单调、平滑。

### 2.2 已修问题（脚本/流程）

- **仿真未真正运行**：去掉 `-r` 后，用「play → 立即 pause」激活仿真，否则 multi_step / wrench 不生效。
- **读到的位置是旧的**：不再用 `scene/info` 服务，改为订阅 `dynamic_pose/info`（或 `pose/info`），配合一次 `multi_step: 1` 触发发布，再读位姿。
- **实验可复现**：每轮实验写入 `experiment_log.json`（命令 + sleep），并用 `reproduce_experiment.py` / `reproduce_experiment_timed.py` 按日志复现。
- **时间假阳性**：用 `analyze_timing_filter.py` 按 sim-time 等筛掉纯 timing 导致的差异；复现脚本支持带时间统计的版本。

### 2.3 已发现并处理过的问题

- **Buoyancy 重置 bug**：reset 后浮力失效（模型下沉），已整理成 issue 提交（见 `documents/issue.md` 等）。
- **Determinism 大量 FAILED**：部分来自 SDF 缺陷（如 contact_extra_data 穿地、RTF=0、缺 Physics 插件），已做静态分析并形成排除规则。
- **Force Removal 系统性失败**：撤力后速度反而跳变 4–8 倍，疑似 wrench/clear 未生效或清除时产生异常冲量，尚未提交 issue，需挑典型用例整理。
- **Temporal Monotonicity**：恒力下位移出现突变（倍数跳变），可能与 Force Removal 同源或为求解器/插件问题。

## 三、初筛「可能测出 Gazebo 问题」的实验

### 3.1 初筛脚本

项目根目录下：

```bash
python3 screen_bug_candidates.py /home/liyitao/workspace/meta/test_22
```

- **默认**：结果打印到 stdout，候选条数打印到 stderr。
- **写 CSV**：`--out candidates.csv`（可指定路径，例如写到 test_22 下）。
- **只取前 N 条**：`--top 200`。
- **看排除原因**：`--verbose`。

### 3.2 排除规则（减少已知非 Gazebo 根因）

- 世界名为 `contact_extra_data`、`contact_sensor`（SDF 穿地等已知缺陷）。
- SDF 中 `real_time_factor` 为 0（激活阶段不可控，易误报）。
- SDF 中未包含 `gz-sim-physics-system`（无物理引擎，行为非常规）。

### 3.3 保留规则（建议优先人工复核）

- **Determinism FAILED**：位置差在 0.01m～500m（排除爆炸级与浮点级）。
- **Force Removal FAILED**：撤力后速度跳变。
- **Zero-Input Stability FAILED**：无力时水平漂移（且非上述排除世界）。
- **Temporal Monotonicity FAILED**：位移突变/非单调。
- **Force Isolation FAILED**：全部保留。
